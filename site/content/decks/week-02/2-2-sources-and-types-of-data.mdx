<Slide title="Internal Data: System Footprint" icon="database" key="The Enterprise Resource Planning (ERP) system acts as the central database of a firm's operational footprint.">

<Bullets items={[
  { l: 1, h: "**Accounting Information System (AIS)** — Records transaction cycles (Sales, Purchasing, Payroll)." },
  { l: 2, h: "Contains the General Ledger, Accounts Receivable, and accounts payable detail." },
  { l: 1, h: "**Supply Chain Management (SCM)** — Tracks inventory, purchasing orders, vendor terms, and logistics." },
  { l: 1, h: "**Customer Relationship Management (CRM)** — Manages leads, sales opportunities, and customer profiles." },
  { l: 1, h: "**Human Resource Management (HRM)** — Tracks employee time cards, payroll rates, and benefits." },
  { l: 1, h: "**Enterprise Resource Planning (ERP)** — The unified system integrating all of these subsystems." }
]} />

</Slide>

<Slide title="Internal Data Structure" icon="layers" key="Siloed systems arise organically in business and require translation layers to integrate cleanly.">

<Bullets items={[
  { l: 1, h: "**Organic Growth & Mergers** — Departments choose localized tools, leading to siloed databases." },
  { l: 2, h: "E.g., Salesforce for CRM, NetSuite for AIS, Workday for HR." },
  { l: 1, h: "**Integration Challenge** — Siloed databases often use different primary keys or identifiers." },
  { l: 2, h: "E.g., CRM tracks customer by `Email`, AIS tracks by `Customer_ID`. Direct joins will fail." },
  { l: 1, h: "**Translation Layers** — Modern analytics relies on ETL (Extract, Transform, Load) pipelines to stitch these systems together." }
]} />

</Slide>

<Slide title="External Data: Market Context" icon="database" key="External data provides critical context (market, regulatory, macro) beyond the boundary of the firm.">

<Bullets items={[
  { l: 1, h: "**Banking & Payment Feeds** — Bank statements, credit card processing reports (e.g., Stripe)." },
  { l: 1, h: "**Market & Academic Databases** — Compustat (financial statement database), CRSP (stock returns/volume)." },
  { l: 1, h: "**Macro & Industry Indicators** — Interest rates (Federal Reserve FRED), CPI (inflation indexes), consumer sentiment." },
  { l: 1, h: "**Regulatory & Tax** — IRS tax tables, SEC EDGAR filings, GAAP/IFRS standards." }

</Slide>

<Slide title="Data Source Considerations" icon="layers" key="Analytics requires balancing the absolute control of internal data with the independent veracity of external data.">

<Bullets items={[
  { l: 1, h: "**Internal Data: High Control, Narrow Scope**" },
  { l: 2, h: "Pros: High control over detail, update frequency, and data definition format." },
  { l: 2, h: "Cons: Blind to external competitor moves, market shifts, or macro trends." },
  { l: 1, h: "**External Data: Broader Scope, Lower Control**" },
  { l: 2, h: "Pros: Provides macro context; often carries higher audit reliability (third-party evidence)." },
  { l: 2, h: "Cons: Cleanliness and format are out of your control; collection methods can be opaque." }
]} />

<Slide title="Types of Data" icon="layers" key="Data falls along a spectrum of organization, from rigid tables to free-form text.">

<Bullets items={[
  { l: 1, h: "**Structured Data** — Data adhering to a strict, fixed, predefined model or table structure." },
  { l: 2, h: "Ready-to-query formats: SQL tables, general ledger systems, spreadsheet matrices." },
  { l: 1, h: "**Unstructured Data** — Data that does not conform to a predefined data model." },
  { l: 2, h: "Requires extraction or AI modeling before use: text documents, emails, contracts, images." },
  { l: 1, h: "**Semi-structured Data** — Data containing tags, markers, or hierarchical structures but not fitting a flat table." },
  { l: 2, h: "Self-describing formats: XML, XBRL, JSON, HTML." }
]} />

</Slide>

<Slide title="Structured Data" icon="database" key="Structured data is the foundation of traditional accounting systems, enabling automated calculations.">

<Bullets items={[
  { l: 1, h: "**Tabular Format** — Organizes information into rows (observations) and columns (variables)." },
  { l: 1, h: "**Quantitative & Relational** — Highly suited for mathematical operations and database relations." },
  { l: 2, h: "Can easily link tables together via primary and foreign keys (e.g. general ledger linked to chart of accounts)." },
  { l: 1, h: "**Examples** — Journal entries, trial balances, accounts payable registers, inventory master files." }
]} />

</Slide>

<Slide title="Structured Data Logic: Wide vs. Long" icon="splitSquare" key="How you shape structured data (wide vs. long) dictates how easily it can be computed or scanned.">

<Bullets items={[
  { l: 1, h: "**Wide Format** — Adds information by creating more columns." },
  { l: 2, h: "Each row has a unique identifier, and columns represent distinct variables or time periods." },
  { l: 2, h: "Ideal for presentation and direct human interaction (e.g., balance sheets, multi-year income statements)." },
  { l: 1, h: "**Long Format** — Adds information by stacking rows vertically." },
  { l: 2, h: "Identifiers are repeated, with dedicated columns for the variable name (key) and the value." },
  { l: 2, h: "Ideal for data storage, querying, and computer aggregation (e.g., databases, Tableau, pandas)." }
]} />

</Slide>

<Slide title="Wide vs. Long" icon="splitSquare" key="Wide format is optimal for human reporting, while long format is designed for computer processing.">

<TwoColumn ratio="1-1">
  <div slot="left">
    <p><strong>Wide Format (Human-Friendly)</strong></p>
    <DataTable
      columns={["ID", "First Name", "Last Name", "# Sales"]}
      rows={[
        ["1", "Aryll", "Zelda", "394"],
        ["2", "Byrne", "Yunobo", "604"],
        ["3", "Cia", "Xani", "12"],
      ]}
    />
  </div>
  <div slot="right">
    <p><strong>Long Format (Machine-Friendly)</strong></p>
    <DataTable
      columns={["ID", "Variable", "Value"]}
      rows={[
        ["1", "First Name", "Aryll"],
        ["1", "Last Name", "Zelda"],
        ["1", "# Sales", "394"],
        ["2", "First Name", "Byrne"],
        ["2", "Last Name", "Yunobo"],
        ["2", "# Sales", "604"],
        ["3", "First Name", "Cia"],
        ["3", "Last Name", "Xani"],
        ["3", "# Sales", "12"],
      ]}
    />
  </div>
</TwoColumn>

<Bullets items={[
  { l: 1, h: "**Wide:** Each observation is one row. Columns represent variables." },
  { l: 1, h: "**Long:** Each observation can span multiple rows. Variables are listed as values in a column." }
]} />

</Slide>

<Slide title="Wide Format Trade-offs" icon="splitSquare" key="Wide formats are intuitive for spreadsheets but scale poorly as new dimensions are added.">

<Bullets items={[
  { l: 1, h: "**Pros: Visually Skimmable** — Humans can easily scan rows and columns without writing filters." },
  { l: 1, h: "**Pros: Dense Data** — Highly efficient when there is little to no missing data across columns." },
  { l: 1, h: "**Cons: High Structural Cost** — Adding a new dimension (e.g. Sales by Month) requires adding 12 new columns." },
  { l: 1, h: "**Cons: Schema Rigidity** — Modifying columns requires altering the table's structure, which can break database queries." }
]} />

</Slide>

<Slide title="Long Format Trade-offs" icon="splitSquare" key="Long formats are standard for databases, facilitating scalable aggregation and multi-dimensional analysis.">

<Bullets items={[
  { l: 1, h: "**Pros: High Scalability** — Adding new dates, divisions, or products only adds rows, not new columns." },
  { l: 1, h: "**Pros: Ideal for Databases** — Extremely efficient for running aggregation queries (sums, averages) in SQL." },
  { l: 1, h: "**Cons: Harder to Scan** — Reading raw records requires scrolling vertically through repetitive labels." },
  { l: 1, h: "**Cons: Requires Pivoting** — Must be reshaped (pivoted) before being fed into certain predictive model matrices." }
]} />

<Quiz
  id="w2-wide-vs-long"
  q="You want to store transaction data that expands daily. If you add a date dimension, which format is more scalable without changing the database columns (schema)?"
  opts={[
    "Wide format (adding a new column for every new date)",
    "Long format (adding new rows with a Date column)"
  ]}
  correct={1}
  why="In a long format, adding a new date simply adds new rows to the existing table structure. Wide format would require adding a new column for every single day, which alters the schema and quickly becomes unmanageable."
/>

</Slide>

<Slide title="Unstructured Data" icon="layers" key="Unstructured data contains the narrative details of business, representing 80-90% of all data generated.">

<Bullets items={[
  { l: 1, h: "**The Narrative of Business** — Contains critical insights that numbers alone cannot capture." },
  { l: 2, h: "Examples: contracts, vendor invoice PDFs, email logs, phone call transcripts, regulatory guidelines." },
  { l: 1, h: "**High Processing Overhead** — Cannot be directly queried with standard SQL or calculated in Excel." },
  { l: 2, h: "Requires translation techniques like Optical Character Recognition (OCR), Text Mining, or Natural Language Processing (NLP)." },
  { l: 1, h: "**AI Revolution** — Large Language Models (LLMs) are dramatically lowering the cost of extracting structure from unstructured documents." }
]} />

<Quiz

</Slide>

<Slide title="Semi-structured Data" icon="layers" key="Semi-structured data uses tags and metadata markup to allow flexible, self-describing structures.">

<Bullets items={[
  { l: 1, h: "**Metadata & Tags** — Employs markers to identify data elements without requiring a rigid column grid." },
  { l: 1, h: "**JSON (JavaScript Object Notation)** — The standard for web API data transmission." },
  { l: 2, h: "Stores data as key-value pairs (e.g. `\"ticker\": \"AAPL\", \"revenue\": 90700000000`)." },
  { l: 1, h: "**XBRL (eXtensible Business Reporting Language)** — The SEC reporting standard." },
  { l: 2, h: "Uses tags like `<us-gaap:CashAndCashEquivalents>` to make financial statements computer-readable." }
]} />
